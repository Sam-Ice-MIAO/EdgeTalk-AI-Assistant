# Agent / RAG 效果评估

## 1. 评估目标

本次评估主要比较 EdgeTalk 中三个接口的回答效果：

| 接口            | 作用                     |
| ------------- | ---------------------- |
| `/chat`       | 普通文本问答，直接调用本地 LLM      |
| `/rag-chat`   | 先检索本地知识库，再让 LLM 基于资料回答 |
| `/agent-chat` | 先判断任务类型，再选择本地工具并生成回答   |

评估重点：

* 回答是否准确；
* 是否基于项目资料；
* 是否出现编造；
* 是否调用了正确工具；

---

## 2. 测试问题

| 编号 | 问题                  | 测试目的              |
| -- | ------------------- | ----------------- |
| 1  | EdgeTalk 当前实现了哪些功能？ | 测试项目事实类问答         |
| 2  | EdgeTalk 后续计划有哪些？   | 测试 RAG 对项目计划的检索效果 |
| 3  | 知识库里有哪些文档？          | 测试 Agent 工具调用     |
| 4  | 这个项目现在做到哪一步了？       | 测试 Agent 项目状态查询   |
| 5  | 请介绍一下你自己            | 测试普通文本问答          |

---

## 3. 测试结果记录

### 3.1 `/chat`

测试问题：

```text
EdgeTalk 当前实现了哪些功能？
```

测试结果：

`/chat` 能正常回答，但回答主要依赖本地 LLM 和 Prompt，本身不会主动检索知识库。对于项目事实类问题，回答可能比较泛，容易遗漏已经完成的能力，例如 ASR、TTS、RAG、Agent 等模块。

结论：

```text
/chat 适合普通开放问答，但不适合作为项目事实类问题的主要接口。
```

---

### 3.2 `/rag-chat`

测试问题：

```text
EdgeTalk 当前实现了哪些功能？
EdgeTalk 后续计划有哪些？
```

测试结果：

`/rag-chat` 能返回 `retrieved` 检索结果，并基于本地知识库内容生成回答。对于“当前功能”和“后续计划”这类项目相关问题，回答比 `/chat` 更准确。

观察结果：

* 能检索到 `current_features.txt`、`project_plan.txt` 等知识库文件；
* 能结合检索内容回答当前功能；
* 能回答后续计划，如 RAG、Agent、多轮对话、Docker、Jetson 等方向；
* 相比 `/chat`，回答更接近项目真实状态。

结论：

```text
/rag-chat 更适合项目知识问答，能降低模型凭空编造的风险。
```

---

### 3.3 `/agent-chat`

测试问题：

```text
知识库里有哪些文档？
EdgeTalk 当前实现了哪些功能？
这个项目现在做到哪一步了？
```

测试结果：

`/agent-chat` 能根据问题选择不同工具。

| 问题                  | 使用工具                   | 结果         |
| ------------------- | ---------------------- | ---------- |
| 知识库里有哪些文档？          | `list_knowledge_files` | 能列出本地知识库文件 |
| EdgeTalk 当前实现了哪些功能？ | `search_knowledge`     | 能调用知识库检索工具 |
| 这个项目现在做到哪一步了？       | `get_project_status`   | 能返回项目当前状态  |

观察结果：

* Agent 能根据关键词完成基础工具路由；
* 对于知识库文件列表这类确定性任务，更适合直接返回工具结果；
* 对于项目知识类问题，Agent 会调用 `search_knowledge`，本质上复用了 RAG 检索能力；
* 对于项目状态类问题，Agent 会调用 `get_project_status`。

结论：

```text
/agent-chat 的价值不在于替代 RAG，而在于根据用户问题选择合适工具。
```

---

## 4. 三类接口对比

| 接口            | 优点            | 局限                 | 适合场景              |
| ------------- | ------------- | ------------------ | ----------------- |
| `/chat`       | 简单直接，响应普通问题   | 不检索资料，项目事实类问题可能不准确 | 普通问答              |
| `/rag-chat`   | 基于知识库回答，事实性更强 | 只能做知识库检索，不会选择其他工具  | 项目资料问答            |
| `/agent-chat` | 能根据任务选择工具     | 当前是规则路由，理解能力有限     | 工具调用、项目状态查询、知识库问答 |

---

## 5. 当前结论

本轮测试说明：

1. `/chat` 能完成基础文本问答，但对项目事实类问题回答较泛；
2. `/rag-chat` 能基于知识库检索结果回答，适合项目知识问答；
3. `/agent-chat` 能根据问题选择工具，适合处理“列文件”“查项目状态”“检索知识库”等任务；
4. RAG 的作用是增强回答依据；
5. Agent 的作用是选择和调用工具；
6. 当前轻量 Agent 方案虽然简单，但逻辑清楚、稳定可控，适合当前项目阶段。
