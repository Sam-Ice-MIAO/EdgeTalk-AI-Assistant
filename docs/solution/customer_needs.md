# 客户需求分析：工业设备维护 AI 助手

## 1. 目标客户

本项目面向制造业、工业自动化和设备运维相关企业，尤其适用于拥有自动化产线、专用设备、检测设备或工业控制设备的客户。

典型客户包括：

- 电子制造企业
- 3C 产品制造企业
- 自动化设备厂商
- 工业软件企业
- 智能制造解决方案服务商
- 工厂设备运维团队

---

## 2. 业务背景

在工业设备维护场景中，现场维修人员经常需要查询设备手册、故障代码表、维修 SOP、巡检清单和安全规范等资料。

这些资料通常分散在不同文档中，且现场人员可能无法长时间打开电脑查找资料。对于新员工来说，故障排查还依赖老师傅经验，处理效率不稳定。

因此，客户希望通过一个本地化 AI 助手，将设备维护知识集中接入，让现场人员能够通过自然语言快速查询相关资料，提高维修效率，降低经验依赖。

---

## 3. 客户痛点

### 3.1 设备资料分散

设备手册、故障代码表、维修 SOP、巡检表、安全规范可能分散在多个文件中，现场人员需要花时间查找和比对资料。

### 3.2 故障处理依赖经验

很多故障需要有经验的人员判断，新员工遇到问题时不知道从哪里开始排查，容易影响维修效率。

### 3.3 现场查询不方便

产线人员可能在设备旁边处理问题，不方便长时间打开电脑或翻阅复杂文档。

### 3.4 企业资料不适合全部上云

设备文档、工艺资料、维护记录可能属于企业内部资料，客户可能更倾向于本地化部署或边缘端部署，减少数据外传风险。

### 3.5 维护过程缺少连续记录

同一次故障排查过程中，维修人员可能会连续询问多个问题。如果系统不能保存会话记录，就难以形成连续的排查上下文。

---

## 4. 业务需求拆解

| 需求方向 | 具体需求 |
|---|---|
| 故障码查询 | 输入故障码后，快速返回故障含义、可能原因和处理建议 |
| 维修 SOP 查询 | 根据维修任务查询标准处理步骤 |
| 点检规范查询 | 查询每日点检、巡检项目和注意事项 |
| 安全规范查询 | 查询设备维护前的安全操作要求 |
| 多轮问答 | 支持同一 session 下的连续追问和历史记录保存 |
| 本地化部署 | 支持在本地环境或边缘设备中运行 |
| API 接口 | 提供接口能力，便于后续接入前端、测试工具或其他系统 |
| 部署验证 | 支持 Docker / MySQL 等方式验证企业部署可行性 |

---

## 5. 业务成功指标

本项目 PoC Demo 的成功标准包括：

1. 是否能快速命中相关设备维护资料；
2. 是否能准确解释常见故障代码；
3. 是否能给出符合 SOP 的处理步骤；
4. 是否能回答点检和安全规范类问题；
5. 是否支持基于 session_id 的多轮维护问答；
6. 是否可以通过 API 方式调用；
7. 是否可以完成本地化或容器化部署验证；
8. 是否为后续边缘设备部署预留空间。

---

## 6. 解决方案方向

EdgeTalk-AI-Assistant 通过 RAG、Agent、Memory、FastAPI 和 Docker 等模块，提供一个工业设备维护 AI 助手 PoC Demo。

整体思路是：

```text
工业知识库
→ RAG 检索相关资料
→ Agent 判断是否调用工具
→ 本地 LLM 生成回答
→ Memory 保存会话记录
→ FastAPI 提供接口
→ Docker / MySQL 验证部署可行性
```

该方案重点验证企业知识库问答在工业设备维护场景中的可行性，而不是直接构建完整生产系统。

---

## 7. PoC 验证范围

当前 PoC 主要验证以下内容：

- 工业知识库问答是否可行；
- 故障码、SOP、点检、安全规范是否能被正确检索；
- Agent 是否能根据问题自动调用知识库工具；
- 会话记录是否能通过 SQLite / MySQL 保存；
- FastAPI 接口是否可以稳定调用；
- Docker 和 MySQL 联调是否可行；
- 语音输入到语音输出的完整链路是否能够跑通。

---

## 8. 当前边界与后续扩展

当前项目仍属于 PoC Demo，重点是验证核心链路，不包含完整生产系统所需的权限管理、前端页面、日志监控、知识库后台管理等能力。

后续可以扩展：

- Jetson Orin Nano 边缘设备部署；
- Linux TTS 替换；
- 实时语音输入；
- 企业文档上传与自动切分；
- 更完整的 PoC 测试报告；
- 前端演示页面；
- 多设备、多用户、多权限管理。

---
