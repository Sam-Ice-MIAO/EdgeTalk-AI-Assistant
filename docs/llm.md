# 本地 LLM 推理测试记录

## 目标

验证本地大模型推理流程：

Text → Local LLM → Reply

本阶段只关注 LLM 模块，不涉及 ASR 和 TTS 串联。

## 实现方式

使用 llama-cpp-python 加载 GGUF 量化模型，在 CPU 上完成本地推理。

## 模型选择

### 第一版：Qwen 0.5B

为了先验证本地 LLM 推理链路是否可运行，我优先选择了参数量更小的 0.5B GGUF 模型。

这样做的原因是：

1. 模型体积更小，下载和加载成本更低；
2. 更容易在 CPU 环境下快速跑通；
3. 适合作为第一版调试模型，用来验证代码、路径和依赖是否正确。

测试结果表明，0.5B 模型可以正常完成文本输入到模型回复的流程，但回答质量和中文表达自然度一般，更适合早期调试，不太适合作为最终 Demo 模型。
### 第二版：Qwen 1.5B

在 0.5B 跑通之后，我进一步测试了 1.5B 模型，希望比较不同参数规模模型在本地语音助手场景中的表现差异。

对比下来，1.5B 模型在以下方面表现更好：

1. 中文表达更自然；
2. 指令理解更稳定；
3. 回答更完整，更适合作为本地语音助手 Demo 的默认模型。

但相比 0.5B，1.5B 的加载和推理速度更慢，对 CPU 资源的要求更高。因此，0.5B 更适合调试，1.5B 更适合展示和后续系统集成。

## 核心参数

- model_path：本地模型文件路径
- n_ctx：上下文长度
- n_threads：CPU 推理线程数
- max_tokens：最大生成长度
- temperature：生成随机性
- stop：控制模型停止生成的位置

## 结论

本地 LLM 模块的实现思路是：先用 0.5B 小模型验证推理流程，再用 1.5B 模型比较效果并做选型判断。

最终我认为：

- 0.5B 模型适合开发初期快速验证链路；
- 1.5B 模型在回答质量上更适合作为语音助手原型的默认模型；
