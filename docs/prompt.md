# Prompt 设计记录

## 1. 目标

Prompt 的目标是让本地 LLM 的回答更加稳定、简洁，并减少重复和编造。

当前 EdgeTalk 主要有两类 Prompt：

* 普通问答 Prompt；
* RAG / Agent 上下文增强 Prompt。

---

## 2. 普通问答 Prompt

普通问答用于 `/chat`，也用于 `/agent-chat` 中不需要调用工具的情况。

当前设计原则：

```text
你是 EdgeTalk，一个本地运行的中文语音助手原型。
你目前主要具备本地文本问答能力，后续会继续接入语音交互和本地知识库问答。
请用简洁、自然、不重复的中文回答用户问题。
回答最多 3 句话，不要编造你不具备的功能。
```

设计目的：

* 限制回答长度；
* 减少重复表达；
* 避免模型编造未实现功能；
* 保持回答风格简洁，适合 Demo 展示。

---

## 3. RAG 问答 Prompt

RAG 问答用于 `/rag-chat`，也用于 `/agent-chat` 中调用知识库检索工具后的回答。

当前设计原则：

```text
你是 EdgeTalk，一个本地运行的中文语音助手原型。
请优先根据给定资料回答用户问题。
如果资料中没有相关信息，请直接说明资料中没有答案，不要编造。
回答要简洁、自然、不重复，最多 3 句话。
```

设计目的：

* 强调基于资料回答；
* 减少模型脱离知识库自由发挥；
* 当资料不足时明确说明没有相关信息；
* 降低项目事实类问题中的幻觉风险。

---

## 4. Agent 工具结果 Prompt

Agent 的流程是：

```text
用户问题
↓
Agent 选择工具
↓
工具返回结果
↓
LLM 基于工具结果生成回答
```

对于不同工具，Prompt 使用策略不同。

### 4.1 确定性工具

例如：

```text
list_knowledge_files
```

这类工具返回的是明确结果，例如文件列表。

对于这种任务，后续更适合直接返回工具结果，而不是让 LLM 重新总结。

### 4.2 知识检索工具

例如：

```text
search_knowledge
```

这类工具返回的是知识库相关文本块，适合交给 LLM 总结成自然语言回答。

### 4.3 项目状态工具

例如：

```text
get_project_status
```

## 5. 推理参数设置

当前本地 LLM 主要使用以下参数：

| 参数               |  当前值 | 作用           |
| ---------------- | ---: | ------------ |
| `max_tokens`     |   80 | 限制回答长度       |
| `temperature`    |  0.3 | 降低随机性，使回答更稳定 |
| `top_p`          | 0.85 | 控制候选词范围，减少发散 |
| `repeat_penalty` | 1.15 | 减少重复表达       |

这些参数主要是为了解决小模型容易出现的几个问题：

* 回答重复；
* 回答过长；
* 表达发散；
* 编造项目未实现功能。

---

## 6. 已观察到的问题

在测试过程中观察到：

1. `/chat` 对项目事实类问题回答较泛；
2. `/rag-chat` 的回答更依赖知识库内容，事实性更好；
3. `/agent-chat` 能选择工具，但工具结果如果再交给 LLM，有时会出现轻微发挥；
4. 文件列表等确定性任务更适合直接返回工具结果；
5. 当前知识库内容较少，RAG 效果受文档质量影响较大。

---

## 7. 当前 Prompt 优化原则

当前项目遵循以下 Prompt 设计原则：

```text
1. 回答要短，避免长篇发散；
2. 回答要基于资料，避免凭空编造；
3. 资料不足时明确说明不知道；
4. 项目功能描述必须符合真实实现；
5. 确定性工具结果尽量直接返回；
6. 检索类结果再交给 LLM 总结。
```
